# FacebookMsg 代码锐评（回滚基线版）

## 总评
这套代码功能是有的，但几个核心工程点不够扎实：同步语义、同步粒度、稳定性护栏。结果是“能用但不稳，能看但不准”。

## 核心问题

### P0 同步语义误导用户
现状：评论中心按钮叫“同步最新数据”，后端默认只拉 20 条帖子。
问题：这会让用户误以为是全量最新，实际只是截断窗口。
结果：用户会认为系统漏抓。

### P1 手动回复后触发全局重同步
现状：回复一条评论后，调用的是全量同步。
问题：路径太重，延迟高、配额消耗高。
结果：响应变慢，用户体验变差。

### P1 缺少单帖同步能力
现状：没有单帖同步入口，运营排障必须跑全量。
问题：精细化运营成本高。
结果：定位慢、资源浪费。

### P1 同步并发缺少护栏
现状：帖子媒体+评论都直接大规模 gather。
问题：帖子数量多时会瞬时并发过大。
结果：更容易触发 429、超时和偶发失败。

### P2 外部 API 瞬态失败无重试
现状：429/5xx 直接失败。
问题：网络抖动下成功率差。
结果：稳定性不够。

## 已按锐评落地修复
1. 同步接口支持全量模式，并将评论中心主按钮明确为“全量同步所有帖子”。
2. 增加单帖同步接口与页面按钮。
3. 手动回复后优先只同步所属帖子。
4. 同步流程增加分批并发执行，降低瞬时压力。
5. Facebook 请求增加指数退避重试（429/5xx）。

## 后续建议
1. 增加同步任务进度条和失败明细。
2. 增加同步链路指标：帖子数、评论数、失败率、重试次数。
3. 为 SyncService 增加分页与异常路径测试用例。
